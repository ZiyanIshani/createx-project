# AI-Powered Technical Due Diligence

Automated technical due diligence for M&A transactions targeting the middle market (deals under $100M). Analyzes software repositories and produces risk reports covering contributor continuity, architectural fragility, code provenance, technical debt, and hidden subscription costs — fully offline by default, optionally powered by the Groq LLM API.

## What It Does

The pipeline runs in up to four stages against any local git repository:

**Step 1 — Repository Ingestion**
- Contributor statistics: commit counts, lines added, activity timelines, recency scoring
- Bus factor analysis: identifies files touched by only one contributor
- Commit velocity: monthly commit frequency and code churn (lines added/removed per month)
- Language detection across all tracked files

**Step 2 — Static Analysis**
- Tree-sitter AST parsing extracts imports and function calls per file
- Dependency graph construction using NetworkX
- Risk metrics: circular dependencies, fragile hubs, orphaned files, external package exposure
- Architectural risk score (0–10)
- Heuristic test coverage estimation via dependency graph cross-referencing
- C-specific semantic analysis: pointer-returning functions, dangerous libc calls (e.g. `gets`, `strcpy`)

**Step 3 — Technical Debt Scoring** *(always runs)*
- Composite 0–100 debt score with five weighted components:
  - Knowledge concentration (30%): bus factor + top-contributor dominance
  - Test coverage gaps (25%): missing tests = fragile future changes
  - Code churn (25%): hot files touched >2× the median are unstable
  - Hub fragility (15%): heavily-imported files are risky to change
  - Documentation density (5%): low comment ratio = high onboarding cost
- Deterministic remediation time estimate (e.g. "2–4 months") based on score and codebase size
- When `--llm` is enabled, a plain-English narrative paragraph is generated by the LLM

**Step 4 — Subscription Service Detection** *(always runs, no LLM needed)*
- Heuristic scan of every tracked file for references to 50+ external SaaS APIs, cloud platforms, payment processors, and data services
- Three signal types: import/package names, URL patterns, and string/comment usage patterns
- Categories covered: Cloud (AWS, GCP, Azure), Payments (Stripe, Braintree), Monitoring (Datadog, Sentry), AI/ML (OpenAI, Anthropic), Auth (Auth0, Okta), and more
- Produces a deduplicated summary grouped by category with reference counts and first-seen location per service

**Step 5 — LLM Agentic Analysis** *(optional, requires Groq API key)*
- **Authorship risk**: who wrote the critical files and are they still around?
- **Provenance risk**: does any code look copy-pasted from online sources?
- **Code quality**: does the code meet your coding standards?

The LLM stage uses a hand-rolled ReAct agent loop (Reason → Act → Observe) against the Groq API and focuses on the top N most fragile/high-risk files.

## Installation

All code lives under `due_diligence/`. Install from there:

```bash
cd due_diligence
pip install -r requirements.txt
```

**Dependencies:**
| Package | Version | Purpose |
|---------|---------|---------|
| `pygit2` | `>=1.14.0` | Git history and blame without subprocess calls |
| `networkx` | `>=3.0` | Dependency graph construction and analysis |
| `tree-sitter` | `>=0.21.0,<0.22.0` | AST parsing |
| `tree-sitter-languages` | `>=1.10.0` | Pre-built language grammars |
| `matplotlib` | `>=3.7.0` | Contributor ↔ file graph visualization |
| `flask` | `>=3.0` | Web dashboard server |
| `requests` | `>=2.31.0` | HTTP calls to Groq API (LLM stage only) |
| `pytest` | `>=7.0.0` | Test suite |

## LLM Backend (Optional)

This tool uses the **Groq API** for LLM analysis. Groq offers a free tier with generous limits (30 RPM, 14,400 RPD).

```bash
export GROQ_API_KEY="your-key-here"
```

Get a free key at: https://console.groq.com/keys

**Recommended models** (pass via `--model`):

| Model | Notes |
|-------|-------|
| `llama-3.1-8b-instant` | Default — fast, free tier |
| `llama-3.1-70b-versatile` | Higher quality, slower |
| `mixtral-8x7b-32768` | Good for long files, 32k context |

## Usage — CLI

```bash
cd due_diligence
python main.py <repo_path> [options]
```

**Arguments:**

| Argument | Default | Description |
|----------|---------|-------------|
| `repo_path` | *(required)* | Path to the local git repository to analyze |
| `--ref REF` | `HEAD` | Git ref (branch, tag, commit SHA) to analyze |
| `--output FORMAT` | `json` | `json` (pipe-friendly) or `pretty` (human-readable) |
| `--llm` | off | Enable LLM agentic analysis via Groq API |
| `--standards PATH` | built-in | Path to a coding standards file (markdown or plain text) |
| `--model NAME` | `llama-3.1-8b-instant` | Groq model name |
| `--top-n N` | `3` | Number of critical files to send to LLM analysis |

**Examples:**

```bash
# JSON output (pipe-friendly), no LLM
python main.py /path/to/repo

# Human-readable report, no LLM
python main.py /path/to/repo --output pretty

# Full analysis with LLM
python main.py /path/to/repo --llm --output pretty

# Custom standards file, specific model, analyze top 5 files
python main.py /path/to/repo --llm --standards standards/my_standards.md \
  --model mixtral-8x7b-32768 --top-n 5 --output pretty

# Analyze a specific branch
python main.py /path/to/repo --ref main --output pretty
```

## Usage — Web Dashboard

Instead of terminal output, view the report as an interactive web dashboard:

```bash
cd due_diligence
python server.py [<repo_path_or_url>] [options]
```

**Arguments:**

| Argument | Default | Description |
|----------|---------|-------------|
| `repo_path` | *(optional)* | Local path or GitHub URL. Omit to use the landing page |
| `--port PORT` | `8080` | HTTP port (avoid 5000 on macOS — reserved for AirPlay) |
| `--ref REF` | `HEAD` | Git ref to analyze |
| `--llm` | off | Enable LLM analysis |
| `--top-n N` | `3` | Critical files to analyze with LLM |

```bash
# Launch with landing page (interactive repo selection)
python server.py

# Launch directly on a specific repo
python server.py /path/to/repo --port 9000

# Clone and analyze a GitHub repo (shallow clone, depth 500)
python server.py https://github.com/org/repo --llm

# Analyze this repo itself
python server.py ..
```

Then open **http://localhost:8080** in your browser.

The dashboard displays: language breakdown, top contributors with activity badges, bus factor risk, dependency graph metrics, architectural risk score, technical debt score, subscription services detected, commit velocity chart (Chart.js), and a Cytoscape.js interactive contributor ↔ file ownership graph.

Click **Re-run** to re-analyze without restarting the server.

### Reference tables

**Recency scores**

| Score | Meaning |
|-------|---------|
| `1.0` | Last commit within 6 months — active |
| `0.5` | Last commit 6–18 months ago — semi-active |
| `0.0` | Last commit older than 18 months — inactive |

**Authorship risk levels**

| Level | Meaning |
|-------|---------|
| `low` | Multiple active contributors |
| `medium` | Few contributors or mixed activity |
| `high` | All contributors semi-active or inactive |
| `critical` | Sole contributor is inactive (knowledge loss risk) |

**Quality grades**

| Grade | Weighted error count (errors + 0.5 × warnings) |
|-------|---------------------------------------|
| A | 0 |
| B | 1–2 |
| C | 3–5 |
| D | 6–10 |
| F | 11+ |

**Technical debt score components**

| Component | Weight | Signal |
|-----------|--------|--------|
| Knowledge concentration | 30% | Bus factor + top-contributor commit share |
| Test coverage gaps | 25% | `100 − coverage_percent` |
| Code churn | 25% | Fraction of files touched >2× median commit count |
| Hub fragility | 15% | Max in-degree relative to codebase size |
| Documentation density | 5% | Comment/docstring line ratio vs 15% target |

## Coding Standards

Place custom standards files in `due_diligence/standards/` (markdown or plain text). Pass the path with `--standards`:

```bash
python main.py /path/to/repo --llm --standards standards/my_standards.md
```

If no standards file is provided, the agent applies built-in defaults:
- Max 50-line functions
- Docstring requirements for all public functions/classes
- No bare `except` clauses
- No magic numbers
- Consistent naming conventions (snake_case, PascalCase, UPPER_SNAKE_CASE)

## Design Constraints

| Constraint | Detail |
|------------|--------|
| **Minimal network** | Network calls only to Groq API when `--llm` is used; everything else is offline |
| **No code execution** | Never compiles, runs, or installs the target repository's code |
| **No code storage** | File content read for LLM analysis is never written to disk |
| **Graceful degradation** | Handles empty repos, binary files, and parse errors without crashing; unavailable Groq API produces a partial result with a warning |
| **Performance** | Steps 1–4 complete in under 60 seconds for repos up to 50k LOC (without LLM) |
| **LLM is opt-in** | `--llm` flag required; without it the full pipeline still runs (Steps 1–4 above) |
| **Subscription scan always runs** | Heuristic only, no API key needed |
